# 对话运行状态与草稿恢复

## 文档状态

- 状态：`DESIGN_READY`
- 负责人：Codex
- 创建日期：2026-07-20
- 最近更新：2026-07-20

## 背景

对话页已经支持 SSE 流式回复、停止任务、刷新后重新监听以及结构化人工交互，但运行结束仍依赖前后端各自推断：正常任务只更新数据库状态，不保证输出 `done`；前端遇到干净的 SSE 结束时又会直接把本地状态改成 `completed`。网络断开、任务刚好结束或页面刷新时，这两套推断可能互相矛盾。

Composer 在任务运行中允许输入文本，但 Enter 会直接停止任务，且未发送文本只存在组件内存。用户容易误停止当前任务，刷新或切换会话后草稿也会丢失。

## 目标

- 每个非等待态运行都产生可消费的终止事件，数据库状态仍是最终事实来源。
- SSE 非预期或无终止事件结束后，前端重新读取服务端状态；运行仍未结束时自动恢复监听。
- 刷新运行中的会话后，从最后一个已知输出事件继续监听，不重复展示已有事件。
- 运行中 Enter 不再隐式停止任务，停止只能通过明确的停止按钮触发。
- 按会话保存纯文本草稿；刷新或切回会话后可以恢复，发送成功后清除。

## 功能范围

- Agent 正常完成和取消时的 `done` 终止事件。
- AgentService 在任务已经结束后仍能排空最后的终止事件。
- chat、resume、interaction resolve 三条 SSE 路径统一执行服务端状态校准。
- 现有空监听流继续使用输出事件游标重连和事件 ID 去重。
- Composer 运行中键盘行为和按会话文本草稿恢复。
- 对上述状态转换和交互增加自动化测试。

## 非功能范围

- 本阶段不实现“当前回复完成后自动发送”的持久化消息队列。
- 不把运行中新消息直接写入现有 Redis 输入流；现有实现会打断当前 flow，不符合下一条消息语义。
- 不持久化尚未发送的附件、上传中的本地文件或技能选择。
- 不修改数据库结构，不引入跨进程任务恢复，不改变人工交互审批规则。
- 不扩展消息编辑、重新生成、分支或模型切换。

## 业务流程

1. 用户发送消息，前端进入流式状态，服务端启动或复用任务。
2. Agent 正常完成时先把 Session 更新为 `completed`，再写入并输出 `done`；等待人工交互时仍以 `wait/interaction` 结束本轮流。
3. 前端收到 `done/error/wait` 时按事件收敛本地状态。
4. 如果 SSE 在没有明确终止事件时结束，前端清理当前连接并刷新 Session：
   - 服务端为 `completed`：恢复 Composer 发送能力；
   - 服务端为 `waiting`：保持 Composer 禁用，由 InteractionCard 接管；
   - 服务端为 `running`：从最后一个已知输出事件恢复空监听流。
5. 任务运行中用户可以继续编辑文本；Enter 作为普通换行，不会停止任务。用户只能点击停止按钮终止任务。
6. 文本变化时按 Session ID 保存草稿；会话切换或刷新时恢复；消息被提交后删除该会话草稿。

## 核心规则

1. `Session.status` 是跨连接的权威状态；SSE 事件用于低延迟投影，不替代断流后的服务端校准。
2. `done/error/wait` 是本轮流的终止事件；Agent 正常完成必须产生 `done`。
3. 读取输出流不能仅以 `task.done` 作为循环入口条件，否则任务先结束时会漏掉已经写入的终止事件。
4. `waiting` 绝不能被通用断流清理覆盖为 `completed`。
5. 停止属于显式破坏当前运行的动作，不绑定到无修饰 Enter。
6. 草稿以用户尚未提交的纯文本为边界，不自动执行、不跨会话串用。

## 现有实现分析

### 相关代码与文档

- `api/app/core/agent/agent_task_runner.py`：消费 Redis 输入流并运行 Agent；正常结束只更新状态，取消时才输出 `done`。
- `api/app/services/agent_service.py`：`while task and not task.done` 读取输出流；任务先结束时可能遗漏尾部事件。
- `web/src/composables/useSessionDetail.ts`：维护消息流、空监听流、事件去重和 Session 本地状态；当前 `finishRunStream()` 会在断流时乐观写入 `completed`。
- `web/src/components/chat/ChatComposer.vue`：运行中 Enter 触发 `stop`，发送按钮被停止按钮替换。
- `web/src/components/chat/ChatInput.vue`：草稿只保存在组件内存，成功提交后清空。
- `docs/debug/chat-composer-stuck-after-stream-end.md`：记录了正常 SSE 结束后本地 `running` 残留的历史问题。

### 可复用能力

- `lastEventId` 与 `seenEventIds` 已能支持输出流续读和前端去重。
- `refresh()` 已同时刷新 Session、事件与文件，可以作为权威状态校准入口。
- `startEmptyStream()` 与指数退避重连已经存在。
- `sessionId` 已传入 ChatInput，可直接作为草稿命名空间。

### 当前约束

- Task 注册表仅存在于当前进程，服务重启后不能恢复原运行。
- Redis 输入流中的运行中新消息会让 AgentTaskRunner 在下一个事件点跳出当前 flow，属于“打断/转向”，不是“下一轮排队”。
- chat POST 同时承担入队和 SSE 订阅，运行中再次调用会替换前端连接，并混用输入流与输出流事件 ID。
- 当前真实浏览器、PostgreSQL 与 Redis 环境不可用，最终端到端门禁只能保留为阻塞项。

## 可选方案

### 方案 A：前端继续推断终态并保存本地草稿

- 实现方式：保留现有后端，只修正 Composer 键盘行为并使用 localStorage 保存文本。
- 优点：改动最小，无后端影响。
- 缺点：断流和任务完成的竞态仍存在，刷新后仍依赖数据库轮询收敛。
- 风险：再次出现“页面显示完成但服务端仍运行”或反向卡住。

### 方案 B：终止事件契约 + 权威状态校准 + 文本草稿

- 实现方式：Agent 始终输出终止事件；服务端排空尾部事件；前端断流后刷新 Session，必要时恢复监听；运行中 Enter 不停止；按会话保存文本草稿。
- 优点：修复状态契约根因，不引入数据迁移；与现有 SSE/刷新能力兼容；可以独立验证。
- 缺点：尚不提供自动发送的下一条消息队列。
- 风险：输出流读取循环需要超时检查，避免异常任务没有终止事件时永久挂起。

### 方案 C：同时实现后端持久化下一条消息队列

- 实现方式：新增独立消息提交接口、客户端消息幂等键、队列状态与取消规则；运行结束后服务端原子领取下一条消息。
- 优点：完整支持刷新、重连和多端下的可靠排队。
- 缺点：需要公共接口、持久化模型、并发幂等和队列管理 UI，交付面明显扩大。
- 风险：若没有原子领取与幂等键，断线重试会重复执行用户消息。

## 方案对比

| 维度 | 方案 A | 方案 B | 方案 C |
| --- | --- | --- | --- |
| 实现复杂度 | 低 | 中 | 高 |
| 状态可靠性 | 低 | 高 | 高 |
| 维护成本 | 低但遗留竞态 | 中 | 高 |
| 向后兼容 | 高 | 高 | 中 |
| 测试难度 | 低 | 中 | 高 |
| 本阶段交付成本 | 低 | 中 | 高 |
| 主要风险 | 状态误判 | 尾事件排空 | 重复执行与队列并发 |

## 推荐方案

采用方案 B。

它直接修复当前“运行何时真正结束”的契约问题，并把断流后的最终判断交还给服务端状态，同时改善最容易误触的 Composer 行为。方案 A 不能解决根因；方案 C 虽然是最终的下一条消息体验，但需要独立的幂等与持久化设计，和本次可靠性修复绑在一起会扩大回归面。

## 数据结构

不修改服务端持久化结构。

浏览器本地草稿：

| 字段 | 类型 | 必填 | 说明 | 约束/默认值 |
| --- | --- | --- | --- | --- |
| storage key | `string` | 是 | `agentic:chat-draft:<sessionId>` | 每个会话独立 |
| value | `string` | 是 | 未发送的纯文本 | 空文本时删除 key |

## 接口设计

### 现有聊天 SSE

- 输入：接口保持不变。
- 输出：正常完成路径保证包含一个 `done` 事件；等待和失败仍分别以 `wait`、`error` 结束。
- 权限：保持现有会话所有权校验。
- 并发：输出读取在任务结束后继续排空，直到终止事件或确认队列为空。
- 兼容性：只增强终止事件保证，不改变现有事件字段。

### 前端流结束协调

- 输入：当前流种类与结束原因。
- 输出：清理连接状态，刷新 Session；若仍运行则重新监听。
- 幂等：旧连接被 Abort 时不触发校准；事件 ID 集合继续过滤重复事件。

## 错误处理与可观测性

- SSE 无终止事件结束不直接报任务失败；记录流结束并刷新权威状态。
- 刷新失败时保留原 Session 状态和可见错误，按现有退避策略恢复监听。
- 输出流在任务结束且超时未读到事件时退出，并记录缺失终止事件告警，避免永久占用请求。
- localStorage 不可用时静默退化为内存草稿，不阻断输入。

## 迁移与回滚

- 迁移：无数据库迁移；旧会话不受影响；草稿 key 首次输入时创建。
- 回滚：移除终止事件增强和本地草稿逻辑即可；服务端事件结构未变化。

## 风险

| 风险 | 可能性 | 影响 | 缓解措施 | 验证方式 |
| --- | --- | --- | --- | --- |
| 任务结束早于 SSE 消费导致漏尾事件 | 中 | 高 | 任务结束后仍排空输出流并设置有限等待 | AgentService 单元测试 |
| 断流刷新与新事件同时发生 | 中 | 中 | 事件 ID 去重，刷新后按服务端状态决定是否重连 | composable 竞态测试 |
| `waiting` 被误写为完成 | 低 | 高 | 终止事件分支和刷新结果均保留 waiting | 前端回归测试 |
| localStorage 异常阻断输入 | 低 | 中 | 所有读写捕获异常并退化 | 组件测试 |
| 自动队列被误认为已实现 | 中 | 中 | UI 不显示排队承诺，文档明确延期 | 验收检查 |

## 重要假设

- Redis 输出流在当前部署中可在任务完成后短暂读取尾部事件。
- 现有 `event_id` 对输出事件足以支持当前单连接续读；本阶段不扩大运行中多次提交。
- 纯文本草稿满足本阶段刷新恢复目标；附件与技能草稿另行设计。

## 待决策项

无。持久化下一条消息队列已明确拆分为后续独立设计，不阻塞本阶段。

## 验收标准

- [ ] Agent 正常完成、取消和异常分别产生 `done`、`done`、`error`，Session 最终为 `completed`。
- [ ] 即使任务在 SSE 消费循环下一次判断前已经完成，客户端仍可收到尾部终止事件。
- [ ] chat、resume、resolve 的 SSE 无终止事件结束时刷新服务端状态，不再无条件乐观标记完成。
- [ ] 刷新结果仍为 `running` 时，从最后输出事件继续监听；`waiting` 不被覆盖。
- [ ] 任务运行中按 Enter 不会调用停止；停止按钮仍可用。
- [ ] 同一会话刷新后恢复文本草稿，不同会话草稿隔离，成功提交后草稿被清除。
- [ ] 聚焦单元测试、前端完整测试、类型检查/构建和差异检查通过。
- [ ] PostgreSQL、Redis 与真实浏览器恢复后完成运行中刷新、断网恢复、等待交互和停止的手工验收。
