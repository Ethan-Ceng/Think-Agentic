# API Agent 总结与优缺点分析

## 一句话总结

`api` 是 MoocManus 的后端 Agent 编排服务。它基于 FastAPI 提供会话、文件、配置、SSE 流式对话和 VNC 代理接口，在一次用户任务中动态创建或复用沙箱，并通过 `PlannerAgent + ReActAgent` 的组合完成“规划、执行、更新计划、总结交付”的闭环。

## Agent 定位

这个 Agent 更接近一个通用任务执行系统，而不是单轮聊天机器人。它的目标是接收用户消息和附件，拆解成可执行步骤，然后在 Docker 沙箱里使用文件、Shell、浏览器、搜索、MCP 和 A2A 远程 Agent 等工具逐步完成任务，并把执行过程以事件形式流式返回给前端。

它适合处理需要多步推理和真实环境操作的任务，例如信息检索、网页操作、文件处理、数据分析、代码运行、报告生成，以及调用外部专业工具或远程 Agent 协作。

## 核心架构

### 入口层

FastAPI 应用在 `api/app/main.py` 中初始化，统一挂载到 `/api` 前缀。主要能力由 `session`、`file`、`app-config`、`health` 等 controller 暴露。

会话对话入口是 `POST /api/sessions/{session_id}/chat`，返回 SSE 事件流。前端可以持续接收用户消息、标题、计划、步骤、工具调用、等待用户输入、错误和完成事件。

### 服务层

`AgentService` 是对话生命周期的主要协调者。它负责检查会话、创建或复用任务、创建沙箱、获得浏览器实例、写入用户消息事件、启动后台任务，并从任务输出流中读取 Agent 事件返回给 SSE。

`AgentTaskRunner` 是真正运行 Agent 的任务执行器。它负责初始化沙箱、MCP、A2A，处理输入消息，把附件同步到沙箱，运行 Planner-ReAct 流，并把工具结果、消息、文件和截图补充为前端可展示的事件。

### Agent 层

系统内部有两个核心 Agent：

`PlannerAgent` 负责把用户需求转成结构化 `Plan`，并在每个步骤执行后根据结果更新后续计划。它强制使用 JSON 输出，并禁用工具调用，定位比较纯粹。

`ReActAgent` 负责执行每个步骤。它继承通用的 `BaseAgent` 循环，按 LLM 响应调用工具，把工具结果写回记忆，再继续迭代，直到生成步骤结果。它还负责最终总结，并把生成文件作为附件交付。

### Flow 层

`PlannerReActFlow` 把两个 Agent 串起来，状态流转大致是：

`IDLE -> PLANNING -> EXECUTING -> UPDATING -> EXECUTING -> SUMMARIZING -> COMPLETED`

执行逻辑是先生成计划，再取第一个未完成步骤交给 ReAct 执行；每个步骤结束后压缩 ReAct 记忆，再交给 Planner 更新后续计划；所有步骤完成后由 ReAct 汇总最终回答。

### 工具层

Agent 可用工具包括：

- `FileTool`：读取、写入、替换、搜索、查找沙箱文件。
- `ShellTool`：执行命令、读取输出、等待进程、写入输入、终止进程。
- `BrowserTool`：通过 Playwright/Chrome 完成导航、查看页面、点击、输入、滚动、按键、执行 JS 和查看控制台。
- `SearchTool`：基于 Bing 页面解析实现网页搜索。
- `MessageTool`：通知用户或向用户提问并进入等待状态。
- `MCPTool`：动态连接 MCP 服务，把外部工具暴露给 LLM。
- `A2ATool`：发现并调用远程 Agent。

工具通过装饰器声明 OpenAI function schema，`BaseAgent` 会收集所有工具 schema 提供给 LLM。实际执行时会按工具名找到对应工具包，并过滤掉 LLM 幻觉生成的多余参数。

### 事件与持久化

对外输出统一使用事件模型，包括 `message`、`title`、`plan`、`step`、`tool`、`wait`、`error`、`done`。这些事件既写入 Redis Stream 用于流式传输，也写入 PostgreSQL 的 session 记录，便于恢复会话详情。

会话还持久化了 Agent memory、文件列表、沙箱 id、任务 id、最新消息和未读消息数。Agent 的上下文记忆按 `planner` 和 `react` 分开保存，ReAct 执行后会压缩部分浏览器和工具结果，降低上下文体积。

## 主要优点

1. 架构分层清晰。API、Service、TaskRunner、Flow、Agent、Tool、Repository 的边界比较明确，后续替换 LLM、沙箱、搜索引擎或任务队列时有清晰扩展点。

2. Planner-ReAct 组合适合复杂任务。规划器先拆步骤，执行器逐步落地，每步后再更新计划，比单 Agent 一次性执行更能处理长任务、动态结果和失败重规划。

3. 工具体系可扩展。工具 schema 由装饰器声明，本地工具、MCP 工具、A2A 远程 Agent 都可以统一暴露给 LLM，整体方向适合做开放式 Agent 平台。

4. 沙箱隔离意识较强。Shell、文件、浏览器等高风险操作都落在独立 Docker 沙箱内，且支持动态创建和复用沙箱，避免直接操作 API 宿主环境。

5. 前端可观测性较好。计划、步骤、工具调用、浏览器截图、搜索结果、Shell 输出、文件内容等都被建模为事件，前端可以展示 Agent 的执行过程，而不是只看到最终回复。

6. 支持人机协作。`message_ask_user` 可以让 Agent 在缺少输入、需要确认或需要用户接管浏览器时暂停，并通过 `WAITING` 状态恢复任务。

7. 有一定容错设计。LLM 调用、工具调用、沙箱等待、MCP/A2A 初始化、数据库 finalize 都有重试或异常处理；JSON 输出使用 `json_repair` 做容错解析。

## 主要缺点与风险

1. 配置文件存在敏感信息泄露风险。当前 `api/config.yaml` 中包含明文 LLM/MCP 凭据，且该文件在仓库内。即使接口返回时隐藏部分字段，源码仓库中的明文密钥仍然是高风险点，应迁移到环境变量或密钥管理系统，并轮换已暴露密钥。

2. LLM 与工具调用强绑定单一协议。`OpenAILLM` 使用 OpenAI Chat Completions 兼容格式，且关闭并行工具调用。接入其他模型供应商时，如果工具调用、JSON 模式或 reasoning 字段不兼容，可能需要额外适配。

3. Agent 的执行可靠性高度依赖 Prompt 和 JSON 输出。Planner 和 ReAct 都要求模型输出特定 JSON 结构，虽然有 `json_repair`，但语义错误、字段缺失、类型不匹配或计划质量差仍可能导致执行失败。

4. 长任务成本和延迟较高。每个步骤都可能经历多轮 LLM 调用、工具调用、计划更新和记忆持久化；MCP/A2A 初始化也在任务运行前发生，复杂任务的首响应和总耗时可能较长。

5. 搜索能力不够稳定。当前 `SearchTool` 直接解析 Bing HTML，容易受搜索页结构、反爬策略、地区语言和网络状态影响。对事实核查任务来说，搜索结果质量和可用性存在不确定性。

6. 任务注册表是进程内状态。`RedisStreamTask` 使用类变量 `_task_registry` 管理运行中任务；如果 API 多进程、多副本部署，或服务重启，已有 task id 无法跨进程恢复，Redis Stream 只能保留消息，不能恢复 Python 任务对象。

7. 会话事件和记忆以 JSONB 数组持续累积。长期会话、工具密集任务会让单行 session 数据变大，影响查询、更新和存储成本。当前只有局部 memory compact，没有事件归档、分页或分表策略。

8. 沙箱权限仍需更细治理。虽然运行在 Docker 沙箱中，但 Agent 可以安装依赖、执行 Shell、访问网络、读写文件。缺少按用户、按任务、按工具的权限策略、资源配额、网络访问控制和审计策略时，生产环境风险较高。

9. 工具失败后的恢复策略偏弱。工具调用失败会以 `ToolResult(success=False)` 交给 LLM 自行处理，但没有统一的失败分类、可恢复动作、熔断、回滚或用户确认策略。

10. 状态模型有边界复杂性。会话状态、Flow 状态、Task done 状态、Redis 输入输出流、数据库事件之间需要保持一致。代码中已经有 rollback 和取消处理，但断线、重连、并发发送消息、任务取消和服务重启仍是容易出现边界问题的地方。

11. MCP/A2A 初始化和清理上下文较脆弱。代码注释中已经说明 streamable HTTP MCP 对 asyncio task 上下文敏感，必须在同一个 task 中清理。这类约束如果后续重构不小心打破，容易产生资源泄漏或运行时异常。

12. 部分实现仍有重构痕迹。`api/README.md` 描述的目录结构与当前实际目录不完全一致，`app_config` 存在 repository 版与 service 版两套配置读写路径，容易让维护者误判真实调用链。

## 改进建议

1. 立即处理密钥管理。把 `config.yaml` 中的密钥迁移到环境变量或 Secret Manager，仓库只保留示例配置；对已经出现过的 LLM、MCP、COS 等密钥执行轮换。

2. 增强任务恢复能力。将 task registry 从进程内内存迁移到可恢复的任务元数据，或明确限制单实例运行；对服务重启后的 RUNNING/WAITING 会话做恢复或标记失败。

3. 拆分事件存储。将 session events、memories、files 拆到独立表或至少支持分页/归档，避免单个 session 行无限增长。

4. 为工具执行增加权限与策略层。按工具类型增加 allowlist、超时、网络域名限制、文件路径限制、资源配额和审计日志，尤其是 Shell、浏览器和 MCP/A2A。

5. 强化结构化输出校验。对 Planner/ReAct 输出增加更明确的 schema 校验、失败重试提示和降级路径，而不仅依赖 JSON 修复。

6. 替换或增强搜索实现。优先接入稳定的搜索 API，保留 HTML 解析作为降级方案，并把搜索结果来源、时间、URL 访问情况纳入事件。

7. 增加端到端测试。重点覆盖 SSE 对话、等待用户输入后恢复、附件同步、工具事件补全、任务取消、服务重启、MCP 初始化失败和沙箱不可用等场景。

8. 收敛配置读写路径。统一 `FileAppConfigRepository` 与 `AppConfigService` 的职责，确保 API 获取、更新、Agent 运行时读取的是同一套配置语义。

## 综合评价

这个 `api` Agent 的整体方向是完整的：它不是简单封装 LLM，而是把会话、计划、执行、工具、沙箱、事件流和持久化组合成了一个可交互的通用 Agent 后端。它的优势在于任务执行链路完整、工具面广、前端可观测性好，并且已经考虑到人机协作和上下文压缩。

主要短板集中在生产化可靠性和安全治理上，包括明文密钥、进程内任务状态、长会话数据膨胀、工具权限边界、搜索稳定性和复杂状态恢复。如果目标是内部 Demo 或单实例试运行，当前架构已经具备较强能力；如果目标是多人、多租户或公网生产服务，建议先补齐密钥管理、权限策略、任务恢复和事件存储治理。
